Django + Inertia + React
vs. Aparte Django & React apps

Wat is het verschil, waarom bestaat er meer dan één aanpak, en wanneer kies je wat?

💡 Eerst een analogie

Stel je voor dat je een restaurantkeuken bouwt. Je hebt een kok (Django, de backend) en een ober (React, de frontend). Het verschil zit in hoe ze met elkaar communiceren:

Bij aparte apps werken de kok en ober in twee losstaande gebouwen. Ze bellen elkaar op via een vaste taal (een API). Dat geeft vrijheid, maar je hebt twee keukens, twee telefoonlijnen, en twee aparte huurcontracten.

Bij Inertia werken ze in hetzelfde gebouw, maar elk met eigen gereedschap. De kok (Django) beslist altijd wát er op het bord komt; de ober (React) beslist hoe mooi het gepresenteerd wordt — zonder aparte telefoonlijn.

Één applicatie. Django stuurt de data direct door naar React via Inertia. Geen aparte API nodig — React krijgt data alsof het een gewone Django-template is.

🔌 Wat is een API eigenlijk? ▼

API staat voor Application Programming Interface — een afgesproken manier waarop twee programma's met elkaar praten. In dit geval: React vraagt aan Django "geef me de lijst van gebruikers", en Django antwoordt met data in JSON-formaat (een gestructureerde tekst, vergelijkbaar met een Python dictionary).

# Voorbeeld van wat Django terugstuurt als JSON
[
  {"id": 1, "naam": "Anna", "rol": "engineer"},
  {"id": 2, "naam": "Bob", "rol": "analyst"}
]

React ontvangt dit, en bouwt er een mooie tabel of grafiek van. Het nadeel: React moet altijd eerst vragen voordat het iets kan tonen. Dit heet een extra round-trip.

🔗 Wat doet Inertia precies? ▼

Inertia is een lijmlaag tussen Django en React. Normaal moet je kiezen: ofwel je bouwt een klassieke Django-app met HTML-templates, ofwel je bouwt een aparte React-app die via een API communiceert.

Inertia geeft je het beste van beide werelden: je schrijft Django-views zoals je gewend bent (met data meegeven), maar in plaats van een HTML-template geef je een React-component terug.

# Django view MET Inertia — geen API nodig
def gebruikers_pagina(request):
  gebruikers = Gebruiker.objects.all()
  return render_inertia('GebruikersLijst', {
    'gebruikers': list(gebruikers.values())
  })

React krijgt de gebruikers-lijst direct mee — geen extra fetch-call nodig. Inertia regelt de communicatie onzichtbaar op de achtergrond.

⚠️ Wat is CORS en waarom is het een probleem bij aparte apps? ▼

CORS (Cross-Origin Resource Sharing) is een beveiligingsmaatregel in browsers. Als je React-app op localhost:3000 draait en je Django-API op localhost:8000, weigert de browser standaard dat React data ophaalt bij Django — want dat zijn twee verschillende "origins".

Je moet Django dan expliciet vertellen welke domeinen toegang mogen hebben. Dit klinkt simpel, maar in productie (met HTTPS, subdomains, en load balancers) is dit een veelvoorkomende bron van frustratie.

Met Inertia heb je dit probleem niet, omdat alles op hetzelfde domein draait.

📦 Wat is het verschil in deployment (live zetten)? ▼

Bij aparte apps heb je minimaal twee processen die draaien en onderhouden moeten worden: een Django-server (bijv. Gunicorn) én een statische hosting voor je React-build (bijv. een S3-bucket of Nginx). Je hebt ook twee CI/CD pipelines nodig.

Bij Inertia bouw je React-bestanden en zet je die neer naast Django. Django serveert alles. Eén server, één pipeline, één configuratie. Voor een data-team zonder DevOps-specialisten is dit een significant voordeel.

Wanneer kies je wat?
✓ Kies Inertia als...
  • Je een intern dashboard of dataportal bouwt
  • Je een klein team hebt (1–3 mensen)
  • Je geen aparte mobiele app nodig hebt
  • Je snel wil kunnen deployen, zonder DevOps-overhead
  • Je team al Django kent en React wil leren
  • Je authenticatie simpel wil houden
→ Kies aparte apps als...
  • →Je bouwt ook een mobiele app die dezelfde data gebruikt
  • →Je hebt een apart frontend-team dat autonoom wil werken
  • →De API moet ook door externe partijen worden gebruikt
  • →Je verwacht dat de frontend en backend in de toekomst heel anders schalen
  • →Je hebt al DevOps-capaciteit in het team
📌 TL;DR — de korte versie

Inertia is minder werk, minder infrastructuur, en makkelijker te begrijpen voor een team zonder front-end specialist. Het is ideaal voor interne tools en dashboards — precies wat data engineers vaak bouwen.

Aparte apps geven meer vrijheid en zijn beter geschikt als je product groeit, meerdere clients (web + mobiel) bedient, of grote teams hebt die onafhankelijk willen werken. De prijs is meer complexiteit.
